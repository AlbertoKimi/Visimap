01

Introducción

VisiMap es una aplicación web de tipo SPA (Single Page Application) desarrollada para digitalizar el registro de visitantes del museo MUVI de Villafranca de los Barros. El sistema permite registrar, consultar y analizar la afluencia de visitantes organizados por provincia o país, integrando un mapa interactivo de España, un calendario de eventos, gráficos estadísticos y un sistema de roles y permisos.

ℹ️
Repositorio GitHub: https://github.com/AlbertoKimi/Visimap.git — Rama activa: Area_de_trabajo

Problemática detectada

📝
Error humano y datos duplicados
El registro manual en papel genera errores de conteo, datos duplicados y resultados inconsistentes que deben revisarse manualmente.
⏱️
Retraso en el análisis
Los informes en papel se demoran un día o más. VisiMap genera estadísticas en tiempo real, acelerando la redacción de informes.
📊
Falta de visualización gráfica
En papel no se aprecian tendencias. Con VisiMap se puede ver de un vistazo qué meses han tenido más afluencia.
🔒
Seguridad y privacidad
Los datos en papel ocupan espacio físico y son vulnerables. VisiMap almacena todo en base de datos con control de acceso.

Capacidades principales

  • Gestión de accesos seguros: Autenticación de usuarios con Supabase Auth y control de roles (administrador / trabajador).
  • Registro inteligente de visitas: Mapa interactivo de provincias y formulario lateral para visitantes internacionales.
  • Análisis estadístico temporal: Gráficas de evolución por días y meses usando Recharts, para identificar picos de actividad.
  • Administración de personal: CRUD de empleados y gestión de perfiles desde el panel de administrador.
  • Trazabilidad completa: Historial de entradas de datos auditable y exportable.

Arquitectura "Front-only"

VisiMap adopta una arquitectura Front-only, delegando completamente el backend a Supabase (BaaS — Backend as a Service). Esto elimina la necesidad de gestionar servidores propios y simplifica enormemente el despliegue y mantenimiento.

Arquitectura de la Aplicación

Frontend

🔵 Tecnologías usadas

⚛️ React 18 ⚡ Vite 5 🎨 Tailwind CSS 🧩 shadcn/ui 🔢 Recharts ✏️ Lucide React 🗺️ React Leaflet 📅 date-fns

React 18 actúa como librería núcleo para la interfaz gracias a su modelo declarativo y basado en componentes. Se hace uso intensivo de Hooks:

  • useState — gestión del estado local (formularios, toggles, datos de visitas)
  • useEffect — llamadas a Supabase al montar componentes y suscripciones en tiempo real
  • useMemo — memoización de cálculos estadísticos costosos para evitar re-renders innecesarios
  • useContext — estado global de autenticación (AuthContext) accesible en toda la app

Vite se utiliza como herramienta de construcción en lugar del tradicional Webpack. Su uso de ESM (ES Modules) nativos en el navegador durante desarrollo permite un arranque instantáneo y un HMR (Hot Module Replacement) extremadamente rápido.

Tailwind CSS con el approach de utility-first permite estilos inline semánticos y carga únicamente el CSS que se usa mediante su proceso de purge en producción. Se complementa con los componentes de shadcn/ui para elementos complejos (modales, dropdowns, toasts).

JSX — Hook personalizado useVisitors 
// src/hooks/useVisitors.js
import { useState, useEffect, useMemo } from 'react';
import { supabase } from '../lib/supabaseClient';

export function useVisitors(month, year) {
  const [visitors, setVisitors] = useState([]);
  const [loading, setLoading]   = useState(true);
  const [error, setError]       = useState(null);

  useEffect(() => {
    async function fetchVisitors() {
      setLoading(true);
      const { data, error } = await supabase
        .from('visitors')
        .select('*')
        .gte('visit_date', `${year}-${month}-01`)
        .lte('visit_date', `${year}-${month}-31`)
        .order('visit_date', { ascending: true });

      if (error) setError(error.message);
      else setVisitors(data);
      setLoading(false);
    }
    fetchVisitors();
  }, [month, year]);

  // Total de visitas por provincia, memoizado
  const statsByProvince = useMemo(() =>
    visitors.reduce((acc, v) => {
      acc[v.province] = (acc[v.province] || 0) + v.count;
      return acc;
    }, {}),
  [visitors]);

  return { visitors, statsByProvince, loading, error };
}

Backend — Supabase (BaaS)

🟢 Tecnologías usadas

☁️ Supabase 🐘 PostgreSQL 15 🔌 PostgREST 🔐 Supabase Auth 🔒 Row Level Security 📦 @supabase/supabase-js

Supabase actúa como BaaS (Backend as a Service), proporcionando una base de datos PostgreSQL gestionada, una API RESTful automática mediante PostgREST, autenticación con JWT y políticas de seguridad a nivel de fila (RLS).

✅
La arquitectura Front-only elimina la necesidad de un servidor Node/Express propio. Toda la lógica de negocio se delega a las políticas RLS de PostgreSQL y a las Edge Functions de Supabase cuando es necesario.
JS — Inicialización del cliente Supabase 
// src/lib/supabaseClient.js
import { createClient } from '@supabase/supabase-js';

const supabaseUrl  = import.meta.env.VITE_SUPABASE_URL;
const supabaseKey  = import.meta.env.VITE_SUPABASE_ANON_KEY;

if (!supabaseUrl || !supabaseKey) {
  throw new Error('Faltan variables de entorno de Supabase');
}

export const supabase = createClient(supabaseUrl, supabaseKey, {
  auth: {
    persistSession: true,        // Sesión persistente en localStorage
    autoRefreshToken: true,      // Renovación automática del JWT
    detectSessionInUrl: true,    // Para OAuth (si se añade en el futuro)
  },
});
JSX — Consulta a Supabase con manejo de errores 
// Registrar una nueva visita desde el mapa interactivo
async function registerVisit(province, count, visitDate) {
  const { data: { user } } = await supabase.auth.getUser();

  const { data, error } = await supabase
    .from('visitors')
    .insert([{
      province,
      count,
      visit_date: visitDate,
      registered_by: user.id,
    }])
    .select()
    .single();

  if (error) {
    console.error('Error al registrar visita:', error.message);
    throw error;
  }

  return data; // Devuelve el registro creado con su id
}

Descripción de las tablas y decisiones de diseño

El modelo relacional de VisiMap ha sido diseñado teniendo en cuenta la naturaleza pública del museo y los dos casos de uso principales: registrar afluencia de visitantes (nacionales e internacionales) y gestionar el personal autorizado. A continuación se explica cada tabla y la razón por la que fue creada de esa forma.

📋 profiles

¿Qué almacena? Los datos del personal del museo: nombre, apellidos, email, teléfono, avatar, rol asignado y si la cuenta está activa.

¿Por qué así? Supabase gestiona la autenticación en su tabla interna auth.users, pero esa tabla no es accesible directamente desde el frontend ni permite añadir campos personalizados. Por eso se crea profiles como extensión: su id es una clave foránea que referencia a auth.users.id, de modo que cada perfil está vinculado a una cuenta de autenticación real. El campo role_id apunta a la tabla roles y es lo que determina qué puede hacer cada usuario en la aplicación.

🏷️ roles

¿Qué almacena? El catálogo de roles posibles en la aplicación: admin y worker.

¿Por qué así? Separar los roles en una tabla propia permite escalar el sistema en el futuro añadiendo nuevos perfiles (por ejemplo, un rol de supervisor) sin tocar la lógica de profiles. Además, al ser una clave foránea, la base de datos garantiza integridad referencial: no puede existir un perfil con un rol que no esté registrado en esta tabla.

📊 registro_visitante

¿Qué almacena? Cada registro de entrada de visitantes: cuántos vinieron, de dónde (provincia o país), cuándo y quién lo apuntó.

¿Por qué así? Es la tabla central de la aplicación y está diseñada para manejar los dos escenarios posibles: visitantes nacionales (con id_provincia) y visitantes internacionales (con id_pais). Ambos campos son opcionales individualmente —uno de los dos siempre tendrá valor—, lo que permite diferenciar la procedencia con una simple comprobación de nulos desde el frontend. El campo tipo_visita permite categorizar el tipo de entrada. El campo id_usuario deja trazabilidad de quién registró cada dato, fundamental para auditorías.

🗺️ provincia

¿Qué almacena? El catálogo de las 52 provincias españolas, con su código ISO y las coordenadas geográficas de su centro.

¿Por qué así? PostgreSQL soporta el tipo point de forma nativa, lo que permite almacenar las coordenadas de cada provincia como un par de valores geoespaciales. Esto es crítico para el mapa interactivo: cuando se hace clic en una provincia, la aplicación puede calcular distancias, centrar la vista o pintar marcadores usando directamente los datos almacenados en la base de datos, sin necesidad de un fichero externo adicional.

🌍 pais

¿Qué almacena? El catálogo de países del mundo con su nombre y código ISO.

¿Por qué así? El MUVI recibe visitantes de fuera de España. Para poder estadificar la procedencia internacional sin crear ambigüedad con los registros nacionales, se optó por una tabla de países separada. El código ISO (bpchar) permite identificar países de forma estándar y podría utilizarse en el futuro para mostrar banderas o integrar datos con APIs externas de turismo.

📅 evento

¿Qué almacena? Los eventos del calendario del museo: exposiciones, visitas guiadas, talleres o cualquier actividad relevante.

¿Por qué así? Uno de los objetivos de VisiMap es poder correlacionar los picos de afluencia con eventos concretos. Al tener los eventos en la base de datos con fecha de inicio y fin, el frontend puede cruzarlos con los registros de visitas y detectar, por ejemplo, que en la semana de una exposición especial la afluencia aumentó un 40%. El campo tipo usa un tipo enumerado (tipo_evento) para categorizar las actividades de forma consistente.

⚙️ configuracion

¿Qué almacena? Parámetros de configuración del sistema en formato clave-valor, junto con quién los modificó por última vez y cuándo.

¿Por qué así? En lugar de incrustar los parámetros del sistema directamente en el código, se guardan en base de datos para que el administrador pueda modificarlos desde la propia aplicación sin necesidad de un nuevo despliegue. El campo actualizado_por (UUID → profiles) garantiza que siempre queda registro de quién cambió cada parámetro, manteniendo la trazabilidad del sistema.

Políticas de Seguridad — Row Level Security (RLS)

Supabase permite definir políticas de seguridad a nivel de fila directamente en PostgreSQL. En VisiMap todas las tablas tienen RLS activado, lo que garantiza que cada petición desde el cliente solo accede a los datos permitidos según el rol del usuario autenticado.

🔍 SELECT — Lectura
Todos los usuarios autenticados pueden leer registros de visitas, provincias y países. Solo los administradores pueden ver todos los perfiles de usuario.
➕ INSERT — Creación
Cualquier usuario autenticado puede insertar registros de visitas. La política verifica que id_usuario coincida con el usuario de la sesión activa (auth.uid()).
✏️ UPDATE — Edición
Los trabajadores solo pueden editar sus propios registros. Los administradores pueden editar cualquier registro o perfil de usuario.
🗑️ DELETE — Eliminación
Únicamente los administradores tienen permisos de eliminación sobre registros de visitas y gestión de usuarios.
⚠️
Las políticas RLS son la primera línea de defensa de seguridad en la arquitectura Front-only. Aunque la clave anon de Supabase es pública, sin las políticas correctas ningún dato sería accesible desde el cliente.

¿Por qué el administrador crea los usuarios?

En VisiMap los trabajadores no pueden registrarse por su cuenta. Esta decisión de diseño está motivada por el entorno de uso: el museo MUVI es una institución con personal controlado, y permitir el autorregistro supondría un riesgo de seguridad al dejar cuentas sin rol asignado o sin supervisión.

El flujo de alta de usuarios es el siguiente: el administrador accede al panel de administración, introduce los datos del nuevo trabajador (nombre, email, contraseña temporal) y le asigna el rol correspondiente. Internamente, esto utiliza la función Admin API de Supabase (supabase.auth.admin.createUser()), que solo está disponible con la clave de servicio (service_role), la cual nunca se expone al cliente. Esto garantiza que únicamente los administradores puedan crear y gestionar cuentas.

ℹ️
La opción de registro público está desactivada en la configuración de Supabase Auth del proyecto (Email Signup: disabled). Solo es posible crear usuarios desde el panel de administración de VisiMap o directamente desde el dashboard de Supabase.

Estado actual: VisiMap aún no está desplegado en producción. La opción prevista es Vercel, por su integración directa con GitHub y su soporte nativo para SPAs. El despliegue se llevará a cabo una vez la aplicación esté completamente terminada. Esta sección se actualizará con la URL de producción cuando esté disponible.

¿Por qué Vercel?

Vercel es la plataforma de despliegue elegida para VisiMap por varias razones técnicas que se alinean perfectamente con la arquitectura del proyecto:

🔗 Integración con GitHub
Vercel se conecta directamente al repositorio de GitHub. Cada push a la rama principal despliega automáticamente una nueva versión sin intervención manual.
⚡ CDN Global
Los archivos estáticos generados por Vite se distribuyen desde la red de servidores de Vercel repartidos por el mundo, reduciendo la latencia para cualquier usuario.
🌐 Soporte SPA nativo
Vercel gestiona automáticamente el redireccionamiento de rutas de React Router sin necesidad de configurar archivos .htaccess ni reglas de servidor.
🆓 Plan gratuito
El plan Hobby de Vercel es gratuito e incluye dominios con HTTPS, despliegues ilimitados y previews por rama, más que suficiente para un proyecto como VisiMap.

Pasos previstos para el despliegue en Vercel

1
Crear cuenta en Vercel
Registrarse en vercel.com usando la cuenta de GitHub para vincular automáticamente los repositorios disponibles.
2
Importar el repositorio
Desde el dashboard de Vercel, seleccionar "Add New Project" e importar el repositorio AlbertoKimi/Visimap desde GitHub.
3
Configurar variables de entorno
En el panel Settings → Environment Variables, añadir las claves de Supabase: VITE_SUPABASE_URL y VITE_SUPABASE_ANON_KEY con los valores del proyecto.
4
Configurar el build
Vercel detecta automáticamente que es un proyecto Vite. El comando de build (npm run build) y el directorio de salida (dist) se configuran solos.
5
Desplegar
Pulsar "Deploy". Vercel construye la aplicación y la publica en un dominio tipo visimap.vercel.app. Desde ese momento, cada nuevo commit en main actualiza la aplicación automáticamente.
⚠️
Recuerda que las claves de Supabase deben configurarse como variables de entorno en Vercel y nunca incluirse directamente en el código fuente ni subirse al repositorio público.

Propuestas de Mejoras

Mejoras aplicadas durante el desarrollo

  • ✅
    Memoización de estadísticas: Se añadió useMemo al hook de estadísticas para evitar recalcular los totales por provincia en cada render.
  • ✅
    División de bundles (code splitting): Configuración de manualChunks en Vite para separar vendor, supabase y recharts en chunks independientes, mejorando el tiempo de carga inicial.
  • ✅
    Persistencia de sesión: Se habilitó persistSession: true en el cliente Supabase para que el usuario no tenga que iniciar sesión cada vez que abre la aplicación.
  • ✅
    Feedback visual inmediato: Se incorporaron toasts de notificación (shadcn/ui Toast) para confirmar o informar de errores al usuario tras cada acción.
  • ✅
    Validación de formularios: Se añadió validación en el cliente antes de llamar a Supabase para evitar peticiones innecesarias a la API.
  • ✅
    Diseño responsive completo: Se adaptó toda la interfaz para funcionar correctamente en pantallas móviles y tablets gracias a las utilidades de Tailwind CSS.
  • ✅
    Protección de rutas por rol: El componente ProtectedRoute se mejoró para verificar no solo la sesión sino también el rol del usuario antes de permitir el acceso.
